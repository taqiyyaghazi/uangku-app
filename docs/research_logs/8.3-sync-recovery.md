# Research Log - Story 8.3: Instant Sync & Recovery (Data Restoration)

## Overview

The goal is to implement a seamless data restoration process when a user logs in on a new device or has an empty local database but existing data in the cloud.

## Key Findings

### 1. Drift Batch Inserts

Drift provides a robust `batch` API that is significantly faster for mass inserts.
Important: We must use `mode: InsertMode.insertOrReplace` (Upsert) to prevent duplication if a partial sync happened before or if the user re-installs.

### 2. Dependency Ordering (Foreign Keys)

Drift enforces foreign key constraints by default (unless disabled, which we shouldn't).
Order of operations:

1. `Categories` (Transactions depend on this)
2. `Wallets` (Transactions depend on this)
3. `Transactions`

### 3. Entry Point & Trigger

The most natural place for "Instant Sync" is upon entering the `DashboardScreen` for the first time after a login.
Logic:

- `DashboardScreen` watches a `syncStatusProvider`.
- `syncStatusProvider` checks if the local database is "new" (e.g., has no categories).
- If new and user is authenticated, it triggers `syncFromCloud()`.

### 4. UI Feedback

We will use a fixed overlay or a "Restoring..." banner at the top of the `DashboardScreen`.
A simple `Stack` in the `DashboardScreen` root or the `AuthWrapper` could handle this.

## Proposed Logic for `syncFromCloud()`

```dart
Future<void> syncFromCloud() async {
  final uid = _getUserId();
  if (uid == null) return;

  // 1. Fetch all from cloud
  final categories = await _syncService.fetchCategories(uid);
  final wallets = await _syncService.fetchWallets(uid);
  final transactions = await _syncService.fetchTransactions(uid);

  // 2. Batch insert to local
  await _db.batch((b) {
    b.insertAll(_db.categories, categories, mode: InsertMode.insertOrReplace);
    b.insertAll(_db.wallets, wallets, mode: InsertMode.insertOrReplace);
    b.insertAll(_db.transactions, transactions, mode: InsertMode.insertOrReplace);
  });
}
```

## Risks & Mitigations

- **Data Volume:** If a user has thousands of transactions, a single `.get()` might be slow or hit memory limits.
  - _Mitigation:_ For V1, a single fetch is acceptable. Future optimization could involve pagination.
- **Foreign Key Failures:** If a transaction in Firestore references a non-existent category/wallet.
  - _Mitigation:_ Ensure `categories` and `wallets` are synced completely before `transactions`. If a reference is missing, Drift might throw. Wrap in try-catch and log to Monitoring.
