# Research Log: Stability & Usage Monitoring (Feature 7.4)

## Topics Investigated

### 1. Firebase Suite Integration in Flutter

- **`firebase_core`**: Foundation for all Firebase plugins.
- **`firebase_crashlytics`**: For crash reporting and non-fatal error logging.
- **`firebase_analytics`**: For usage tracking and screen monitoring.

### 2. Platform Configuration (Android Kotlin DSL)

- The project uses `build.gradle.kts`, which requires specific syntax for adding plugins compared to Groovy.
- Root `build.gradle.kts`: Add classpath in `plugins` block with `apply false`.
- App `build.gradle.kts`: Apply the plugin in the `plugins` block.

### 3. Screen Tracking in Flutter

- Since the app uses `MaterialApp` with `home: MainShell()`, automated screen tracking can be enabled by adding `FirebaseAnalyticsObserver` to the `navigatorObservers` property of `MaterialApp`.

### 4. Global Error Handling

- Fatal errors: `FlutterError.onError` and `PlatformDispatcher.instance.onError`.
- Non-fatal errors: `FirebaseCrashlytics.instance.recordError`.

## Findings

### Package Selection

- `firebase_core: ^3.11.0`
- `firebase_crashlytics: ^4.3.0`
- `firebase_analytics: ^11.4.0`

### Implementation Strategy

1.  **Dependency Addition**: Update `pubspec.yaml`.
2.  **Android Setup**: Add Crashlytics plugins and Google Services.
3.  **Initialization**: Update `main.dart` with Firebase initialization and global error catchers.
4.  **Analytics Wrapper**: Create `AnalyticsService` in `lib/core/services/` to provide a clean API for custom events.
5.  **Integration**:
    - Inject `AnalyticsService` into relevant providers (like `ExportNotifier`).
    - Add `FirebaseAnalyticsObserver` to `MaterialApp`.

## Architecture Decisions (ADR Summary)

- **Service Layer**: We'll use a `Feature` based approach or `Core` service for Analytics. Given it's a cross-cutting concern, a `Core` service is more appropriate.
- **Abstraction**: `AnalyticsService` will abstract the `FirebaseAnalytics` instance, making it easier to swap or mock in tests.
