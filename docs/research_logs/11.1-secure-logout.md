# Research Log: Epic 11 - Secure Logout & Local Data Cleansing (Story 11.1)

## Request Context
- **Epic:** 11 - Data Privacy & Session Security
- **Story:** 11.1 Secure Logout & Local Data Cleansing
- **Goal:** Wipe local databases (Drift) and `SharedPreferences` when the user triggers the logout process.
- **Constraints:** Must be atomic, show non-dismissible loading overlay during cleansing, and require confirmation dialog.

## Current Implementation
- **Local Persistence (Drift):** Located in `lib/data/database.dart`. Contains tables: `Wallets`, `Transactions`, `InvestmentSnapshots`, `AppSettings`, `Categories`, `Budgets`.
- **State Management:** Riverpod.
- **Auth Layer:** Managed by `FirebaseAuthRepositoryImpl` in `lib/features/auth/repository/auth_repository_impl.dart`. Uses `FirebaseAuth` and `GoogleSignIn`.
- **UI Element:** Logout happens via `UserAvatarButton`'s bottom sheet (`_ProfileSheet`) in `lib/features/auth/widgets/user_avatar_button.dart`.

## Technical Plan & Findings
1. **Drift `deleteAllLocalData()`:**
   - Must use Drift's transaction to ensure atomicity.
   - Deleting order is currently handled automatically by SQLite if cascade delete is not strict, but it's best to call `await delete(table).go()` for each table.

2. **Integration into Auth Logic:**
   - The user story explicitly states "Gabungkan logika ini di dalam AuthService".
   - Our system uses `AuthRepository` for Firebase interface.
   - To keep logic self-contained, `FirebaseAuthRepositoryImpl` should receive `AppDatabase` via dependency injection and wipe local persistence and `SharedPreferences` *before* calling `_firebaseAuth.signOut()`.

3. **UI Enhancements:**
   - Needs `showDialog` for the confirmation step: _"Are you sure? This will clear all local data from this device. Your data remains safe in the cloud."_
   - Needs a `showDialog(barrierDismissible: false)` loading overlay while waiting for `repo.signOut()` to finish.

## Open Architecture Questions
- Should we clear `AppSettings` too? If yes, standard settings are reset, which is exactly what AC #2 implies.
- Should this complex sequence be extracted into an `AuthService` Use-Case instead of modifying `AuthRepository` directly? The implementation plan suggests updating `AuthRepositoryImpl` to match the existing architecture.

## Conclusion
The path forward is clear. The primary changes will occur in `database.dart`, `auth_repository_impl.dart`, and `user_avatar_button.dart`. The logic conforms to the requirements outlined in Story 11.1.
