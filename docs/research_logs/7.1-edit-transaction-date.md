# Research Log: 7.1 Edit Transaction Date

## Feature Understanding

The user wants to edit the date of an existing transaction. This requires updating the UI (`TransactionDetailSheet` instead of `EditTransactionSheet`, which is the actual file name) to include a date selector in edit mode, and updating the database.

## Findings

1. **Edit Modal UI:** `lib/features/transaction/widgets/transaction_detail_sheet.dart` contains the view and edit modes. The view mode displays the date, but the edit mode currently lacks a date selector.
2. **Date Selector Implementation:** `lib/features/transaction/widgets/quick_entry_sheet.dart` has an `ActionChip` below the amount for selecting the date. We will replicate this UI in `TransactionDetailSheet`'s edit mode.
3. **Time Retention:** When changing the date, we must retain the original transaction's time (`tx.date.hour`, `tx.date.minute`, `tx.date.second`) to maintain chronological order, as requested in the AC.
4. **Database Update:** The save functionality uses `repo.updateTransactionAtomic`. We'll modify the `TransactionsCompanion` to include `date: Value(_selectedDate)`.
5. **Dashboard Synchronization:** Drift's reactive streams (`watchDailySpending`, `watchMonthlySummary`, etc.) automatically map to changes in the underlying table if queried via `.watch()`. Thus, updating the row's date will push a new state to any Provider listening to these streams, fulfilling AC #3. No explicit "refetch" command is needed since it's driven by Drift `watch()`.
6. **Consistent UI State:** `_selectedDate` will be initialized in `_resetEditState()` using `widget.transaction.transaction.date`.

## UI/UX Specifics

- Add an `ActionChip` with a calendar icon, behaving similarly to `QuickEntrySheet`.
- After successful update, provide visual feedback with a `SnackBar` saying "Transaction date updated" if not already covered by a generic success message.
- Disable future dates (`lastDate: DateTime.now()`).
